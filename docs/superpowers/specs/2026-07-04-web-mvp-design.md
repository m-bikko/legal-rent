# Web MVP «RentLegal KZ» — дизайн-спека

Дата: 2026-07-04 • Статус: утверждена пользователем

## Цель

Веб-приложение с минимальным функционалом платформы аренды недвижимости РК (по `MVP-functions.txt`, в рамках этапа 1 из `TZ_platforma_arenda_kz_v1.1.md`): регистрация/логин по номеру телефона с OTP, роли арендатор/арендодатель, лента объявлений с фильтрами и лайками, детали объекта, верификация арендодателя, публикация объектов, привязка арендатора и подписание договора (упрощённое), статусы объектов.

## Утверждённые решения

1. **Бэкенд:** Supabase (проект `xmkfklhubdbtsocfwloh`, credentials в `.env`). Supabase MCP подключён к другому проекту — не используется; миграции применяются напрямую через Postgres (`POSTGRES_URL_NON_POOLING`).
2. **OTP:** dev-заглушка. Код генерируется, хранится хешем, проверяется с expiry и лимитом попыток; вместо Telegram показывается в dev-панели UI. Интерфейс `OtpSender` — единственная точка замены на Telegram Gateway в будущем.
3. **Структура:** монорепо pnpm workspaces + Turborepo.
4. **Языки:** ru / kz / en с первого дня (next-intl, локаль в URL).

## Архитектура

```
apps/web              # Next.js 15 (App Router), ant-design v5, Tailwind, React Query
packages/core         # типы, zod-схемы, бизнес-логика (KZ-телефоны, статусы, верификация)
packages/i18n         # словари ru/kz/en
```

**Доступ к данным только через сервер.** Браузер не обращается к Supabase напрямую: все чтения/мутации — через Next.js Route Handlers с service-role ключом. Сессия — httpOnly JWT-cookie (jose, подпись `SUPABASE_JWT_SECRET`). RLS включён в режиме «запретить всё» для anon/authenticated — контур безопасности MVP, публичное чтение можно открыть позже.

## Модель данных

| Таблица | Ключевые поля |
|---|---|
| `users` | id uuid PK, phone unique (E.164 +7…), role `tenant\|landlord`, account_type `individual\|self_employed\|organization`, full_name, org_name, iin_bin, city, verification_status `none\|pending\|approved\|rejected`, created_at |
| `otp_codes` | id, phone, code_hash, expires_at, attempts int, consumed bool, created_at |
| `verification_requests` | id, user_id FK, type `self_employed\|organization`, data jsonb (ИИН, номер/срок удостоверения, адрес / БИН, название, юр. адрес), doc_paths text[], status `pending\|approved\|rejected`, created_at |
| `properties` | id, owner_id FK, type `apartment\|house\|dacha\|office\|commercial\|building\|space`, address, city, gis_url, price numeric, rent_period `hour\|day\|month`, description, photos text[], contact_phones text[], whatsapp_phones text[], status `free\|rented\|archived`, created_at |
| `favorites` | user_id + property_id PK |
| `rental_agreements` | id, property_id FK, landlord_id FK, tenant_id FK (найден по номеру), status `draft\|active\|ended`, landlord_signed_at, tenant_signed_at, created_at |

**Storage:** `property-photos` (public read), `verification-docs` (private, доступ только через сервер).

**Правила статусов:** объект `rented` ⇔ существует agreement со статусом `active` (обе подписи). Расторжение/архив возвращает `free`/`archived`. Публикация объектов доступна только `verification_status = approved`.

## Auth-флоу

- **Регистрация:** выбор роли («Хочу арендовать» / «Я арендодатель») → switch типа аккаунта (арендодатель: Самозанятый | ИП/ТОО; арендатор: Физлицо | ИП/ТОО) → форма (ФИО или ИИН/БИН+название, город, номер с маской +7 7XX XXX XX XX) → OTP → сессия.
- **Логин:** номер → OTP → сессия.
- OTP: 6 цифр, TTL 5 минут, максимум 5 попыток, повторная отправка с кулдауном.

## Экраны (mobile-first)

- **Шелл:** нижний menu-bar у арендатора (Объявления • Сохранённые • Профиль), у арендодателя + вкладка «Мои объекты». Desktop ≥ md: верхняя навигация.
- **Объявления:** лента карточек (карусель фото, цена, город, тип), селектор города (вкл. «Весь Казахстан»), фильтры (тип, цена от/до, срок аренды), лайк с карточки.
- **Детали:** галерея, цена/срок, описание, адрес, «Позвонить» (tel:) и «WhatsApp» (wa.me), 2ГИС-превью (embed по ссылке, fallback — кнопка «Открыть в 2ГИС»), лайк.
- **Сохранённые:** список лайкнутых.
- **Профиль:** бейдж верификации, данные аккаунта, переключатель языка, выход. Для арендатора — кнопка «Стать арендодателем» → форма верификации (тип указан, поля предзаполнены). Для арендодателя без верификации — форма верификации (самозанятый: ИИН, ФИО*, номер и срок удостоверения, адрес; организация: файл уведомления о начале деятельности, ИИН/БИН*, название*, юр. адрес; * — предзаполнено).
- **Мои объекты:** список с бейджами Свободен/В аренде/В архиве, создание/редактирование объекта (все атрибуты + загрузка фото), привязка арендатора по номеру → договор → кнопки «Подписать» у обеих сторон → объект «В аренде».

## i18n

next-intl, маршруты `/{locale}/…` (ru по умолчанию), все UI-строки в словарях `packages/i18n`, идентичный набор ключей для трёх языков.

## Обработка ошибок

- Единый формат ответа API `{ ok, data | error: { code, message } }`; коды ошибок мапятся на переводимые сообщения.
- Валидация zod на клиенте (формы antd) и повторно на сервере.
- OTP: неверный код / истёк / превышены попытки — отдельные коды ошибок.
- Привязка арендатора: номер не найден → понятная ошибка с подсказкой.

## Тестирование

- Jest + RTL: юнит-тесты `packages/core` (нормализация/валидация KZ-номеров, генерация/проверка OTP, переходы статусов объекта, правила верификации по типу аккаунта).
- Компонентные тесты ключевых форм.
- Финальная проверка: `pnpm build`, прогон основных флоу в браузере (регистрация обеих ролей, публикация, лайк, договор), скриншоты.

## Вне скоупа MVP

Реальный Telegram OTP, платежи, налоговый модуль, ЭЦП/eGov, коммунальные платежи, админ-панель модерации (верификация подтверждается вручную в БД или через временный служебный эндпоинт), посуточное бронирование с календарём.
